# Contributing

Thanks for wanting to contribute.
One rule up front:

**Pull requests targeting `main` must be raised through [`no-mistakes`](https://github.com/kunchenguid/no-mistakes).**
Pushing through it runs an AI-driven review/test/lint pipeline in an isolated worktree, forwards the push only after every check passes, and opens a clean PR automatically.

A GitHub Actions check (`Require no-mistakes`) runs on PRs targeting `main` and fails if the body is missing the deterministic signature that no-mistakes writes. The release and dependency bots are exempt so their automation keeps working; other PRs without the signature will not be merged.

## Repo conventions

- Node 20+, TypeScript, ESM-only.
- Install dependencies with `pnpm install --frozen-lockfile`; run `pnpm run build`, `pnpm run lint`, and `pnpm test` before pushing. `pnpm test` runs vitest with coverage and enforces 100% thresholds (lines, functions, branches, statements).
- `src/supa.ts` is the single boundary to the `supabase` binary and the Management API. Keep all child-process / network calls there so commands stay pure and hermetically testable. Tests mock `../src/supa.js` (commands) or `node:child_process` / `node:fs` / `fetch` (the wrapper) — never touch the real CLI, Docker, or network.
- Do not hand-edit `CHANGELOG.md` or `.release-please-manifest.json` — they are regenerated by [release-please](https://github.com/googleapis/release-please) from your conventional commit messages. A CI check (`Guard generated files`) fails the PR if either is modified; the release bot is exempt.
- Do not hand-edit `skills/supabase-axi/SKILL.md`. It is generated from the shared skill source (`src/skill.ts`); run `pnpm run build:skill` and commit the result. CI fails (`pnpm run build:skill -- --check`) if it drifts.
- `bench/` is a separate, isolated sub-package (its own `package.json`, `pnpm-workspace.yaml`, and toolchain) holding the agent benchmark. It is intentionally excluded from the root `tsconfig`, `eslint`, and `vitest`, so it never affects the main package's build, lint, or 100%-coverage gate. Install and run it from inside `bench/`; do not add it to the root build. See [`bench/README.md`](./bench/README.md).

## Design

supabase-axi follows the [AXI](https://github.com/kunchenguid/axi) (Agent eXperience Interface) standards: TOON output, minimal default schemas, blob previews with line/char counts, pre-computed aggregates, definitive empty states, a content-first home view, contextual next-step suggestions, and structured errors with consistent exit codes (0 success, 1 error, 2 usage). Keep new commands consistent with these principles.

## Workflow

1. Fork the repo and create a branch.
2. Make your changes; add or update tests under `test/`.
3. Run `pnpm run build && pnpm run build:skill && pnpm run lint && pnpm test`.
4. Commit with a [conventional commit](https://www.conventionalcommits.org/) message (e.g. `feat: …`, `fix: …`).
5. Push through the gate instead of pushing to `origin`:

   ```sh
   git push no-mistakes
   ```

6. Run `no-mistakes` to drive the pipeline. Once it passes, it opens the PR against `main` for you with the required signature.
